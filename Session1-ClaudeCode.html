Context Strategies

✓ DO

/clear between unrelated tasks.

/compact proactively — don't wait for the warning.

Be specific about which files to look at.

Break large tasks into smaller conversations.

✗ DON'T

Paste entire files — Claude can read them itself.

One conversation for multiple unrelated tasks.

Forget to clear after a task — stale context causes drift.

Expect Claude to "remember" from 50 messages ago.

What Makes Opus Special?

1M token context — 5x more than Sonnet

That's roughly all 7 Harry Potter books. Opus can hold entire codebases in its head at once.

Extended Thinking

Opus can "think" internally before responding — like a senior engineer who pauses, considers tradeoffs, then gives a well-reasoned answer. Use /effort max to unlock deepest reasoning.

The catch: tokens

Opus 4.6 consumes significantly more tokens than 4.5 did. That "thinking" costs extra. This is why we don't use it for everything — save it for when you need the big brain.

Opus thinking
> "design the auth service migration"
 
⧖ Thinking...
considering backwards compatibility...
evaluating session vs JWT tradeoffs...
checking for breaking changes in API...
 
Here's my recommended migration plan:
Phase 1: Add new auth alongside old
Phase 2: Migrate endpoints gradually
Phase 3: Remove legacy after 2 sprints
 
# Opus considered 3 approaches and
# picked the safest one. Sonnet would
# have jumped straight to code.

CLAUDE.MD: YOUR TEAM'S PLAYBOOK

SQUIRE

What Is Claude.md?

A markdown file at your repo root that Claude reads every time it starts.

Think of it as your team's constitution — coding style, architecture decisions, what to never do.

Version control it. Review changes in PRs like any other code.

Pro Tip

CLAUDE.md costs tokens every session (~500-2K depending on size). Keep it concise — every line you add gets loaded every time. Think of it like a config file, not a wiki.

Non-engineers: you don't edit this file, but it shapes how Claude responds for everyone — terminal and desktop app alike.

# squire-platform
 
## Architecture
Monorepo: apps/web, apps/api, packages/
Stack: React, TypeScript, Express, Prisma
 
## Conventions
- Use Zod for all input validation

SQUIRE

Claude.md Tips

good vs bad CLAUDE.md
# ✗ Too vague — Claude will guess wrong:
Validate inputs.
Use good naming.
Follow best practices.
 
# ✓ Specific — Claude knows exactly what to do:
Use Zod schemas for all API input validation.
Variables: camelCase. Types: PascalCase. Files: kebab-case.
Run `pnpm lint && pnpm test` before committing.
Never use `any` — use `unknown` with type narrowing.

Keep it concise — every line costs tokens every session.

Update it as the project evolves. Stale instructions are worse than none.

Use /init to scaffold one for an existing project.

Plan Mode

Before Claude writes a single line of code — make it plan first.

/plan tells Claude to think about the approach, list the files it will touch, identify risks, and lay out the steps — without making any changes yet.

Why this matters:

Without a plan, Claude jumps straight to code. With a plan, you review the strategy, catch mistakes early, and guide the direction before any code is written.

Best combo: plan with Opus, implement with Sonnet.

Opus thinks deeply about architecture. Sonnet executes quickly. Use each where it shines.

/plan example
# Step 1: Plan with Opus
> /model opus
> /plan
> "migrate auth from sessions to JWT"
 
⧖ Planning...
Plan:
1. Add JWT utils in src/auth/jwt.ts
2. Update middleware (3 files)
3. Migrate /login and /refresh
4. Add tests for token expiry
5. Keep session fallback for 2 weeks
 
# Step 2: Approve, switch, implement
> "looks good, implement it"
> /model sonnet
✓ Implementing Phase 1...

Why Learning This Matters Now

Models are getting smarter — but also more expensive to run.

Opus 4.6 consumes more tokens than 4.5 did. The "thinking" features add overhead. As models improve, knowing how to manage context, pick the right model, and prompt efficiently will matter more, not less.

If you learn to work with models efficiently now:

→ You'll get better results

→ You'll burn fewer tokens

→ You'll be ready when models get even more powerful

Haiku — learn the codebase

Low cost, fast answers

Opus — plan, architect, review

High cost, deep thinking

Sonnet — implement fast

Great balance, your default

The right model for the right job = better output, lower cost.
