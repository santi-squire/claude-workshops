Context Strategies

DO

/clear between unrelated tasks.

/compact proactively -- don't wait for the warning.

Be specific about which files to look at.

Break large tasks into smaller conversations.

Take screenshots of useful Claude outputs -- build your own knowledge base.

DON'T

Paste entire files -- Claude can read them itself.

One conversation for multiple unrelated tasks.

Forget to clear after a task -- stale context causes drift.

Expect Claude to "remember" from 50 messages ago.

Context Budget

CLAUDE.md, system prompt, and tools are "always packed" in every session. Keep CLAUDE.md lean = more room for actual work.

/context
> /context
 
Context usage: 68%
■ system  ■ you  ■ free
 
# Above 80%? Run /compact
# Auto-compact at 95%
Warning

If Claude auto-compacts (at 95%), quality drops. Start a new conversation instead. Think of auto-compact as an emergency brake, not a feature.

Real Case @ Squire

We had 4 memory files eating ~7.7k tokens (3.8%) every session. Merged them into a lean CLAUDE.md → dropped to 4.6k (2.3%). ~40% reduction in auto-loaded docs, freeing up 14k more tokens for work.

Before: 7.7k tokens in memory files

After: 4.6k tokens, 78% free space

Data & Privacy

What Claude CAN see:

Your conversation, files it reads, CLAUDE.md, tool outputs

What Claude does NOT do:

Does not train on your data (Team plan). Conversations are not shared between users. Each session is isolated.

Our policy:

We use Claude Team -- enterprise-grade privacy. Your prompts and code are not used to train models.

SQUIRE

Models & the Ideal Workflow

What Makes Opus Special?

1M token context -- 5x more than Sonnet. Holds entire codebases in its head.

Extended Thinking -- Opus "thinks" internally before responding, like a senior engineer who pauses and considers tradeoffs. Use /effort max to unlock deepest reasoning.

The catch: Opus 4.6 consumes significantly more tokens than 4.5 did. That "thinking" costs extra. Save it for when you need the big brain.

Why This Matters

As models get smarter and more expensive, knowing how to pick the right model, manage context, and prompt efficiently will matter more, not less.

The 3-Step Workflow

1. HAIKU -- Explore

Quick context gathering. "Explain the auth module structure."

↓

2. OPUS -- Plan

Comprehensive planning. "Design the migration with phases and risks."

↓

3. SONNET -- Build

Fast implementation. "Implement Phase 1 from the plan."

Haiku explores → Opus plans → Sonnet builds

This saves tokens AND gets better results than using one model for everything.

SQUIRE

Plan Mode

Before Claude writes a single line of code -- make it plan first.

/plan tells Claude to think about the approach, list the files it will touch, identify risks, and lay out the steps -- without making any changes yet.

Why this matters:

Without a plan, Claude jumps straight to code. With a plan, you review the strategy, catch mistakes early, and guide the direction before any code is written.

Best combo: plan with Opus, implement with Sonnet.

Opus thinks deeply about architecture. Sonnet executes quickly. Use each where it shines.

/plan example
# Step 1: Plan with Opus
> /model opus
> /plan
> "migrate auth from sessions to JWT"
 
⧇ Planning...
Plan:
1. Add JWT utils in src/auth/jwt.ts
2. Update middleware (3 files)
3. Migrate /login and /refresh
4. Add tests for token expiry
5. Keep session fallback for 2 weeks
 
# Step 2: Approve, switch, implement
> "looks good, implement it"
> /model sonnet
✓ Implementing Phase 1...
This is the biggest productivity tip. Plan first, code second.

# squire-platform
 
## Architecture
Monorepo: apps/web, apps/api, packages/
Stack: React, TypeScript, Express, Prisma
 
## Conventions
- Use Zod for all input validation
- camelCase vars, PascalCase types

good vs bad CLAUDE.md
# Bad -- too vague:

Use good naming.
Follow best practices.

Variables: camelCase. Types: PascalCase.

Never use `any` -- use `unknown` + narrowing.
